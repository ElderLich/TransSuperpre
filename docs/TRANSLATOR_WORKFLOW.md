# TransSuperpre Translator Workflow

This project uses local translation workspaces plus GitHub Actions. Translators
edit the card files locally, then upload their workspace files to the GitHub
repo. GitHub does the heavy work: importing new cards, applying mapping fixes,
building the `.ypk`, updating `test-release.json`, and writing dump files.

## Short version

1. Run your local `AutoPR.py` helper.
2. Choose `Refresh local raw2 from latest .ypk`.
3. Translate cards in your local `<LANG>/raw2` folder.
4. Choose `Upload raw2 workspace and push to GitHub`.
5. GitHub Actions updates the repo and builds the package.
6. Check the language folder on `main` for the generated `.ypk`, JSON, and
   version file.

## What each folder means

Inside the GitHub repo, every workspace-based language has this shape:

```text
<LANG>/
  Base Files/
    test-release.cdb
    test-strings.conf
  Workspace/
    test-release.cdb
    test-strings.conf
    Mappings.csv
    Reports/remaining_cn.json
  test-release.json
  version.txt
  ygopro-super-pre.ypk
```

`Workspace` is the translator input folder. This is where translated
`test-release.cdb`, `test-strings.conf`, and `Mappings.csv` go.

`Base Files` is generated by the workflow from the latest upstream Super-Pre
package. Do not translate in `Base Files`; it can be overwritten by automation.

`test-release.json`, `version.txt`, and `ygopro-super-pre.ypk` are release
outputs. Do not edit them by hand.

`Reports/remaining_cn.json` is a helper report from `PromptPatcher.py`. It is
useful for checking untranslated Chinese text or mapping gaps.

The `dump` branch is read-only output for humans. It shows text dumps of CDB
changes so reviewers can see what changed without opening SQLite. Do not edit
the `dump` branch.

## Supported languages

The workspace flow is used by:

- `DE`
- `EN`
- `ES`
- `FR`
- `IT`
- `JP`
- `KR`
- `PT`

`ZH-TW` uses a separate workflow and edits `ZH-TW/Base Files` directly.

## Local setup for translators

Install these first:

- Python 3
- Git
- GitHub Desktop or Git command line
- Access to the TransSuperpre GitHub repo
- A CDB editor such as DataEditorX or DB Browser for SQLite

Each translator can keep their files anywhere. The paths do not need to match
the maintainer's computer.

Recommended local folder shape:

```text
Super-Pre Translations/
  ES/
    AutoPR.py
    Mappings.csv
    raw2/
      test-release.cdb
      test-strings.conf
```

Use the matching language folder name: `DE`, `EN`, `ES`, `FR`, `IT`, `JP`,
`KR`, or `PT`.

Copy `Tools/AutoPR_local.py` from the TransSuperpre repo into your local
language folder and rename it to `AutoPR.py`.

Example:

```text
TransSuperpre/Tools/AutoPR_local.py
  -> Super-Pre Translations/ES/AutoPR.py
```

The helper detects the language from the folder name. If the TransSuperpre repo
is not in a common location, the helper asks for the repo path the first time
and saves it in `.transsuperpre_local.json` beside the language folders.

You can also set one of these environment variables:

```text
TRANSSUPERPRE_REPO=C:\path\to\TransSuperpre
TRANSSUPERPRE_ROOT=C:\path\to\TransSuperpre
SUPER_PRE_LOCAL_ROOT=C:\path\to\Super-Pre Translations
```

## Daily translation process

1. Pull the latest `main` branch in the TransSuperpre repo.
2. Run `AutoPR.py` and choose option `1` to refresh your local `raw2` files.
3. Edit your local `raw2/test-release.cdb`.
4. Edit your local `raw2/test-strings.conf` when prompt strings or archetype
   strings need changes.
5. Edit your local `Mappings.csv` when a repeated Chinese phrase should be
   replaced automatically.
6. Run `AutoPR.py`.
7. Choose option `2` to upload and push.

The refresh option downloads the latest upstream `.ypk`, extracts fresh
`test-release.cdb` and `test-strings.conf`, merges new cards into local `raw2`,
removes stale CDB rows that no longer exist upstream, removes local
`raw2/test-update.cdb` if it exists, and applies the local `Mappings.csv`.

Upload option `2` copies these files into the GitHub repo and pushes them:

```text
raw2/test-release.cdb -> <LANG>/Workspace/test-release.cdb
raw2/test-strings.conf -> <LANG>/Workspace/test-strings.conf
Mappings.csv -> <LANG>/Workspace/Mappings.csv
```

If you prefer GitHub Desktop, choose option `3`. That copies the files into the
repo only, then you can review, commit, and push with GitHub Desktop.

Command-line equivalents:

```powershell
py AutoPR.py refresh
py AutoPR.py upload
py AutoPR.py copy
py AutoPR.py pull
py AutoPR.py status
py AutoPR.py config
```

## What GitHub Actions does

There are three workflow types for each workspace-based language.

### Workspace sync

This runs daily, manually, and when `Mappings.csv` or automation scripts change.

It does this:

1. Downloads the latest upstream `ygopro-super-pre.ypk`.
2. Downloads the live language `cards.cdb`.
3. Extracts fresh upstream `test-release.cdb` and `test-strings.conf` into
   `Base Files`.
4. Adds any new missing cards into `Workspace/test-release.cdb` while keeping
   existing translations.
5. Adds any missing string lines into `Workspace/test-strings.conf`.
6. Applies `PromptPatcher.py` and `Mappings.csv` to reduce repeated manual
   replacements.
7. Removes workspace `test-update.cdb` if it exists, because it is generated
   fresh during release.

### Release build

This runs after a successful workspace sync, manually, and when a translator
pushes `Workspace/test-release.cdb` or `Workspace/test-strings.conf`.

It does this:

1. Copies translated workspace files into `Base Files`.
2. Applies mappings again before packaging.
